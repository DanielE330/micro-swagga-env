# Swagger Aggregator

Легковесный микросервис на FastAPI для объединения OpenAPI (Swagger) документации из разрозненных микросервисов в единый интерфейс. 

Отлично подходит для проектов на Docker Compose или в легких Kubernetes-кластерах, где разворачивать тяжеловесные решения (вроде Kong, KrakenD или Backstage) избыточно.

## Почему это решение работает лучше стандартных

Контейнер `swaggerapi/swagger-ui` умеет принимать список URL, но на практике это часто приводит к двум проблемам: CORS-ошибкам в браузере и неработающим кнопкам "Try it out", потому что сервисы не знают о своих префиксах на шлюзе (API Gateway).

Swagger Aggregator решает эти проблемы архитектурно:

*   **Нет проблем с CORS.** Браузер запрашивает документацию только у агрегатора. Сам агрегатор ходит за спецификациями (`openapi.json`) по закрытой внутренней Docker-сети `server-to-server`.
*   **Динамическая подмена Base URL.** Агрегатор перехватывает JSON-спецификацию и на лету подменяет в ней массив `servers`. Теперь запросы из Swagger UI гарантированно уходят по правильным публичным маршрутам через ваш reverse-proxy.
*   **Language-Agnostic.** Неважно, на чем написаны микросервисы (FastAPI, Express.js, Spring Boot, Go/Swag). Подойдет любой сервис, отдающий валидный `openapi.json`.
*   **Zero-Dependency и минимализм.** Менее 100 строк кода. Не требует монтирования конфигурационных файлов. 

## Быстрый старт

Добавьте сервис в ваш `docker-compose.yml`:

```yaml
  docs_service:
    build: ./docs_service
    expose:
      - "8000"
    environment:
      # формат: хост:порт=префикс_на_шлюзе
      - SERVICES=auth_service:8002=/auth,user_service:8001=/users
```

Пробросьте запросы в вашем балансировщике (nginx, caddy, traefik) на порт этого сервиса. Например, `http://localhost:8080/docs`.

## Конфигурация

Вся магия происходит через одну переменную окружения `SERVICES`.

### Синтаксис переменной SERVICES

Строка, разделенная запятыми. Каждый блок описывает один микросервис:
```text
host:port=prefix,host:port=prefix,...
```

| Параметр | Описание | Пример |
| :--- | :--- | :--- |
| `host` | Имя контейнера во внутренней сети (или IP/домен) | `auth_service` |
| `port` | Внутренний порт, который слушает микросервис | `8002` |
| `prefix`| Публичный роут, по которому микросервис торчит наружу | `/auth` |

**Пример:**
```env
SERVICES=auth:8002=/auth,users:8001=/users,billing:8003=/billing,notify:8005=/notify
```

### Дополнительные настройки

| Переменная | По умолчанию | Описание |
| :--- | :--- | :--- |
| `CACHE_TTL` | `30` | Время (в секундах) кеширования спецификаций. Снижает паразитный трафик между сервисами. |

## Как добавить новый микросервис

Добавление происходит в два шага:

1. Убедитесь, что ваш целевой сервис отдает файл спецификации по пути `/openapi.json` (В FastAPI, например, это работает из коробки).
2. Допишите его конфигурацию в переменную `SERVICES` через запятую:
   ```yaml
   environment:
     - SERVICES=auth:8002=/auth,users:8001=/users,new_service:8003=/new
   ```

Названия сервисов в выпадающем списке Swagger подтягиваются агрегатором автоматически из самих OpenAPI-спецификаций (поле `info.title`).

## Локальный запуск и разработка

Если вам нужно подебажить агрегатор без Docker:

```bash
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# Запуск с моковыми сервисами
SERVICES=localhost:8002=/auth,localhost:8001=/users uvicorn main:app --reload --port 8000
```
